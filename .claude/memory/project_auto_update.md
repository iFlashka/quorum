Auto-update — **сквозная архитектурная тема**, заложенная пользователем в PROJECT.md (секция «Auto-update», стр. 93–113). Реализуется в фазе 7, но конфиги/версионирование закладываются с фазы 0, чтобы не делать breaking-правок инфраструктуры позже.

**Why:** пользователь явно сказал «заложено в архитектуру с начала». Это не косметическая фича — она диктует дисциплину версионирования (`package.json` ↔ `tauri.conf.json`), наличие public-ключа в сборке клиента (если public-ключ появится только в фазе 7, старые клиенты не смогут получать обновления — они уже разойдутся по друзьям), и появление CI-флоу на тегах.

**How to apply:**

## Что закладывается уже в фазу 0

- Версия `0.0.1` синхронно в `apps/desktop/package.json` и `apps/desktop/src-tauri/tauri.conf.json`. CI-проверка совпадения — добавится в фазе 7, но дисциплина с начала.
- Заглушка секции `plugins.updater` в `tauri.conf.json` с **пустым** `pubkey`-плейсхолдером и комментарием «заполнить в фазе 7 после `tauri signer generate`». **Не ставим** сам плагин, не пишем клиентский код апдейтера — это фаза 7.
- ADR `0002-auto-update-architecture.md` с описанием решения и плана.
- Раздел «Auto-update» в README с пометкой «реализуется в фазе 7».

## Что реализуется в фазе 7

- Установка `@tauri-apps/plugin-updater` (JS) + `tauri-plugin-updater` (Rust).
- Однократная генерация пары ключей через `tauri signer generate`.
- Public-ключ → в `tauri.conf.json` → `plugins.updater.pubkey`.
- Private-ключ → в `GitHub Secrets` как `TAURI_SIGNING_PRIVATE_KEY`.
- GitHub Actions на теге `v*.*.*`: сборка `.msi`, подпись, публикация в Release, генерация `latest.json`.
- Endpoint в `tauri.conf.json` → `plugins.updater.endpoints` указывает на `latest.json` в GitHub Releases.
- Клиентская логика: проверка при запуске + раз в час, toast `«Доступно X.Y.Z»`, страница «О программе» с changelog и кнопкой ручной проверки, прогресс-бар скачивания, применение «Сейчас» / «Позже».
- CI-валидация совпадения версий в `package.json` и `tauri.conf.json`.
- Тестовый прогон `0.0.1 → 0.0.2` через локальный endpoint (env-переменная), документировать в README.

## Критические моменты (никогда не игнорировать)

- **Code signing** (Microsoft Authenticode) и **подпись манифеста updater** (Tauri signer) — это **разные вещи**. Code signing мы НЕ делаем (платно, $200–400/год), пользователи увидят SmartScreen — это ок. Подпись манифеста updater — **обязательна**, без неё updater откажется применять обновление.
- **Потеря приватного ключа updater = катастрофа.** Все клиенты, у которых уже зашит соответствующий public-ключ, не смогут принять обновление, подписанное другим ключом. Решение — переустановка с нуля у всех пользователей. **Ключ бэкапить минимум в 2 места** (например, password manager + офлайн-копия), сразу после генерации в фазе 7.
- **Public-ключ в `tauri.conf.json` нельзя «незаметно» поменять задним числом** — при ротации все клиенты, собранные со старым ключом, ломаются.
- **Версии MUST совпадать** в `package.json` и `tauri.conf.json`. CI ловит, но и руками не забывать.
