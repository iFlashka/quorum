# Quorum — самохостимый Discord-аналог для близкого круга (Desktop)

## Контекст и цели

Я создаю self-hosted приложение для общения в стиле Discord под названием **Quorum**. Сервер крутится на VPS (~4 vCPU / 8 GB RAM, Hetzner или аналог), пользователи подключаются через **нативное desktop-приложение под Windows** (Mac/Linux — бонусом, не приоритет). Аудитория — **5–10 близких друзей**.

Это не SaaS-продукт и не масштабируемая платформа. Это пет-проект для своих, где приоритет — **простота развёртывания, надёжность, низкое потребление ресурсов и ощущение «настоящего приложения», а не браузера**.

Проект пишется итеративно, фазами. Каждая фаза должна быть рабочей и тестируемой до перехода к следующей.

## Архитектура верхнеуровнево

Это **client-server** приложение с двумя половинами:

1. **Сервер (бэкенд + media)** — крутится на VPS, обслуживает всех клиентов. Никакого веб-интерфейса для пользователей; на сервере только API + WebSocket + LiveKit + coturn.
2. **Десктоп-клиент** — нативное приложение под Windows на **Tauri**. Внутри — React/TypeScript, но снаружи это полноценное `.exe` с инсталлятором, иконкой в трее, глобальными хоткеями и push-уведомлениями.

Веб-интерфейса как отдельной точки входа НЕТ. Сервер не отдаёт HTML на корне. Всё взаимодействие — через десктоп-клиент. Это упрощает: не нужно поддерживать два UI, не нужно думать про мобильные браузеры, проще auth (можно использовать system keychain).

## Целевой функционал MVP

1. **Авторизация:** регистрация по invite-коду, логин/пароль, JWT-токены. Refresh-токен хранится в системном keychain (Tauri OS keychain plugin), не в localStorage — безопаснее. Никаких email-подтверждений, OAuth, восстановления пароля.
2. **Серверы и каналы:** один или несколько «серверов» (guilds), внутри текстовые и голосовые каналы. Роли: owner / admin / member.
3. **Текстовый чат:** real-time через WebSocket, история, редактирование, удаление, реакции эмодзи, ответы (reply), упоминания @user, базовый markdown, загрузка изображений и файлов (до 25 MB).
4. **Голосовые звонки 1-на-1:** WebRTC peer-to-peer, минимальная нагрузка на сервер.
5. **Групповые голосовые каналы:** до ~8 человек одновременно. SFU через **LiveKit**.
6. **Демонстрация экрана и видео:** через тот же LiveKit-стек. Demo экрана — высокий приоритет, видео — опционально.
7. **Presence:** онлайн / отсутствует / не беспокоить, кто в каком голосовом канале.
8. **Desktop-only фичи (то ради чего мы и взяли Tauri):**
   - **Глобальный push-to-talk** — настраиваемая клавиша (например `Caps Lock`), работает даже когда приложение свёрнуто
   - **System tray** — иконка в трее с правым кликом «открыть», «mute», «выйти»
   - **Native push-уведомления** — через системные нотификации Windows (toast notifications)
   - **Автозапуск с системой** (опционально, в настройках)
   - **Закрытие в трей** — крестик не закрывает, а сворачивает в трей
   - **Auto-updater** — приложение проверяет новые версии и обновляется само

## Что НЕ входит в MVP

- Мобильные приложения (iOS/Android)
- E2E-шифрование сообщений
- Боты, интеграции, webhook-и
- Стримы, форумные каналы, треды
- Веб-версия (только десктоп-клиент)
- Захват системного звука при screenshare (это сложно на Windows, отдельная фаза если понадобится)

- **Zod** для валидации схем
- **pino** для логирования
- **LiveKit Server SDK** для генерации room-токенов

**Realtime media (на VPS, отдельные контейнеры):**
- **LiveKit** (self-hosted) для SFU
- **coturn** как TURN/STUN

**Desktop-клиент:**
- **Tauri 2.x** (актуальная мажорная)
- **React 18 + TypeScript + Vite** (внутри webview)
- **TanStack Router** для роутинга
- **TanStack Query** для серверного стейта
- **Zustand** для клиентского стейта
- **TailwindCSS** + **shadcn/ui** для UI
- **Lucide Icons**
- **LiveKit Client SDK** для голоса/видео/screenshare

**Tauri plugins (важные):**
- `tauri-plugin-global-shortcut` — глобальные хоткеи (push-to-talk)
- `tauri-plugin-notification` — нативные уведомления
- `tauri-plugin-os` — определение платформы
- `tauri-plugin-store` — персистентные настройки
- `tauri-plugin-stronghold` или `tauri-plugin-keychain` — безопасное хранение токенов
- `tauri-plugin-autostart` — автозапуск
- `tauri-plugin-updater` — auto-updates
- `tauri-plugin-window-state` — запоминание размера/позиции окна

**Инфраструктура (только для серверной части):**
- **Docker Compose** для оркестрации сервисов на VPS
- **Caddy** как reverse proxy с автоматическим Let's Encrypt
- **GitHub Actions** для CI: lint, тесты, сборка десктоп-клиента под Windows

**Сборка и distribution:**
- Tauri собирает `.msi` или `.exe` инсталлятор для Windows через GitHub Actions
- Релизы через GitHub Releases
- Подписи кода (code signing) на старте не делаем (платно, $200-400/год), пользователи увидят SmartScreen warning при первой установке — это приемлемо для близких друзей

**Auto-update (важно, заложено в архитектуру с начала):**
- **Updater plugin: `@tauri-apps/plugin-updater`** + Rust-side `tauri-plugin-updater`
- **Источник обновлений:** GitHub Releases (бесплатно, без своего сервера). При каждом релиз-теге GitHub Actions:
  1. Собирает `.msi` инсталлятор под Windows
  2. Подписывает его приватным ключом updater (см. ниже)
  3. Публикует в GitHub Release как asset
  4. Генерирует и публикует `latest.json` манифест с версией, URL инсталлятора и подписью
- **Криптографическая подпись манифеста (обязательна, не путать с code signing):**
  - Один раз сгенерировать ключевую пару командой `tauri signer generate`
  - **Публичный ключ** зашить в `tauri.conf.json` → попадёт в сборку
  - **Приватный ключ** положить в GitHub Secrets как `TAURI_SIGNING_PRIVATE_KEY`
  - При сборке Actions подписывает артефакты автоматически
  - **КРИТИЧНО:** приватный ключ забэкапить в несколько мест. Его потеря означает что все пользователи должны переустановить приложение с нуля — обновления через старую цепочку доверия больше не пройдут.
- **UX обновлений (как у Discord/Slack):**
  - Проверка при запуске + раз в час пока приложение открыто
  - Если найдено обновление → toast в углу: «Доступно обновление X.Y.Z. Перезапустить и установить?» + кнопки «Сейчас» / «Позже»
  - В настройках страница «О программе» с текущей версией, кнопкой «Проверить обновления вручную» и changelog последних версий
  - Скачивание в фоне с прогресс-баром в toast
  - Применение — после клика «Сейчас» или при следующем запуске если выбрано «Позже»
- **Версионирование:** semver (`x.y.z`), версия в `package.json` и `tauri.conf.json` синхронизирована. CI проверяет совпадение.
- **Тестирование:** в README раздел «Как протестировать auto-update локально» — собрать версию `0.0.1`, установить, потом собрать `0.0.2` и подложить локальный манифест через переменную окружения для updater endpoint.

**Тесты:**
- **Vitest** для юнит и интеграционных
- Покрытие критичных частей: auth, права, отправка сообщений, WebSocket-обработчики

## UI / Дизайн-система — Discord-like dark theme

Интерфейс «вдохновлён Discord», но НЕ пиксельный клон. Повторяем информационную архитектуру и UX-паттерны, но не используем логотип, маскота Wumpus, фирменный шрифт gg sans, точные иконки и фирменные иллюстрации.

**Layout (структурно как Discord):**
- Левая узкая колонка (~72px): список серверов — круглые иконки, squircle на ховере, белый «pill» слева как индикатор активного / непрочитанного
- Вторая колонка (~240px): для выбранного сервера — каналы по категориям (текстовые с #, голосовые с иконкой динамика). Снизу — карточка пользователя: аватар, никнейм, кнопки микрофона / наушников / настроек
- Центр: чат канала или UI голосового канала. Сверху хедер с названием и описанием
- Правая колонка (~240px, сворачиваемая): участники по ролям и онлайн-статусу

Цвета — CSS variables, подключены к Tailwind через `theme.extend.colors`. Светлую тему не делаем, но архитектурно готовы добавить.

**Типографика:**
- **Inter** через `@fontsource/inter` — основной шрифт (gg sans проприетарный, не используем)
- 14px — основной текст, 16px — заголовки, 12px — таймстампы
- **JetBrains Mono** для кода

**Окно приложения (Tauri-специфика):**
- Кастомный titlebar (frameless window) с собственными кнопками min/max/close — даёт более «нативный продуктовый» вид как у Discord/Slack
- Минимальный размер окна: 1000×600
- Запоминание размера и позиции между запусками (`tauri-plugin-window-state`)
- Drag region на верхней панели чтобы можно было таскать окно

**Поведенческие паттерны:**
- Hover на сообщении → панель действий справа (реакция / ответить / редактировать / ещё)

DropdownMenu, ContextMenu, Dialog, Tooltip, Popover, ScrollArea, Avatar, Separator, Tabs, Switch, Slider, Sonner. Кастомизируются через CSS-переменные.

## Архитектура — ключевые решения

1. **Монорепа** через **pnpm workspaces**:
   - `apps/server` — Node.js бэкенд
   - `apps/desktop` — Tauri-приложение (внутри `src/` фронт на React, `src-tauri/` Rust-обёртка)
   - `packages/shared` — общие типы, Zod-схемы, API-контракты
2. **Один сервер**, без микросервисов. Логически разделён на модули: auth, guilds, channels, messages, voice, presence, files.
3. **WebSocket для всего realtime, кроме медиа** — текст, presence, typing, события.
4. **Сигналинг WebRTC для 1-на-1** через наш WebSocket. Групповые звонки полностью через LiveKit.
5. **Безопасность:** все эндпоинты под auth (whitelist для login/register), rate limiting на login и регистрацию, sanitize HTML (DOMPurify на клиенте + сервере), refresh-токены в OS keychain.
6. **Файлы:** на старте локальная директория с volume mount, абстрагировано за интерфейсом — потом легко перевести на S3/MinIO.
7. **Tauri IPC:** общение между Rust-частью и React-частью только через типизированные команды (`#[tauri::command]`). Никакого «магического window.something».

## Качество кода

- **TypeScript strict mode** везде, никаких `any` без явного комментария почему
- ESLint + Prettier, конфиг в репо
- API-эндпоинты типизированы end-to-end (общие типы в `packages/shared`)
- Логирование через **pino**, структурированное JSON
- Конфигурация через env, валидируется Zod при старте, есть `.env.example`
- README с разделами: «как запустить локально», «как задеплоить сервер», «как собрать клиент»

## Конфигурация подключения у клиента

Клиент должен знать адрес сервера. Решение:
- При первом запуске экран onboarding: «введите адрес вашего Quorum-сервера» (например `quorum.твойдомен.com`)
- Адрес сохраняется в `tauri-plugin-store`
- Можно сменить через настройки → «Server» → «Disconnect»
- Логин/пароль — отдельный экран после ввода адреса

## План работы — фазы

Я хочу работу по фазам. **Не переходи к следующей фазе пока я не подтвержу, что текущая работает и устраивает.** В конце каждой фазы должен быть рабочий, запускаемый продукт.

**Фаза 0 — Bootstrap**
- Инициализируй монорепу с pnpm workspaces
- Настрой TypeScript, ESLint, Prettier, Vitest
- Подними базовый Fastify-сервер с health-check
- Инициализируй Tauri-проект с React + Vite внутри
- **Заложи дизайн-систему:** Tailwind с CSS-переменными из палитры, Inter, shadcn/ui с кастом-темой, Lucide
- **Свёрстай статичный shell приложения** (без логики): три колонки в стиле Discord, frameless window с кастомным titlebar, заглушечные данные. Я должен сразу увидеть «правильную» визуальную картинку
- `docker-compose.yml` для серверной части с postgres, redis, livekit, coturn (дефолтные конфиги)
- `Caddyfile` для прода
- README с командами `pnpm dev:server`, `pnpm dev:desktop`, `pnpm test`, `pnpm build:desktop`
- **Критерий завершения:** запускаю `docker compose up -d` (сервер) и `pnpm dev:desktop` (клиент), открывается нативное окно с тёмным интерфейсом в стиле Discord

**Фаза 1 — Auth и базовые сущности**
- Схема БД: users, guilds, members, channels, invites
- Drizzle миграции, seed-скрипт для dev
- API: register по invite, login, logout, me, refresh
- JWT (access + refresh), refresh в OS keychain через Tauri-плагин
- Onboarding-флоу клиента: ввод адреса сервера → login/register → главный экран
- Тесты на auth-флоу

**Фаза 2 — Текстовый чат**
- Таблицы messages, attachments, reactions
- WebSocket-протокол: подключение с JWT, подписка на каналы, события `message.create/update/delete`, `reaction.add/remove`, `typing.start`, `presence.update`
- API + UI для каналов, отправки/редактирования/удаления, история с cursor-пагинацией
- Загрузка файлов через multipart, рендер изображений inline
- Markdown через `marked` + DOMPurify
- Реакции, ответы, упоминания
- Typing indicators, presence через Redis

**Фаза 3 — Tray, нотификации, окно**
- System tray иконка с меню (открыть, mute, выход)
- Закрытие крестиком → в трей, не выход
- Native Windows toast notifications для упоминаний и DM (когда окно не в фокусе)
- `tauri-plugin-window-state` — запоминание размера/позиции
- Опция «запускать при старте Windows» в настройках
- Бейдж непрочитанных на иконке трея и в taskbar

**Фаза 4 — Голос 1-на-1 + Push-to-talk**
- WebRTC сигналинг через WebSocket
- UI: «позвонить» в DM, входящий, принять/отклонить
- Управление микрофоном/камерой
- Наш coturn как ICE-сервер
- **Глобальный хоткей push-to-talk** через `tauri-plugin-global-shortcut` — настраиваемая клавиша в настройках, индикатор-оверлей когда нажата
- **Шумодав (как в Discord):** базовый уровень — встроенные WebRTC-флаги `noiseSuppression: true`, `echoCancellation: true`, `autoGainControl: true` в `getUserMedia`. Дополнительно — нейросетевой шумодав через **RNNoise WASM** (open source, без лицензии); тогл «шумодав» в настройках голоса (вкл/выкл/«агрессивный»)

**Фаза 5 — Голосовые каналы (LiveKit)**
- LiveKit Server SDK на бэке (генерация токенов с правами)
- Клиент: вход/выход в голосовой канал, список участников, индикатор speaking, mute/deafen, push-to-talk работает и тут
- **Шумодав в LiveKit-pipeline:** прокидываем тот же RNNoise WASM-обработчик в `LocalAudioTrack` через preprocessor API, чтобы тогл из настроек работал и в групповых каналах
- Тестим на 3+ участниках

**Фаза 6 — Видео и демонстрация экрана**
- Поверх LiveKit: переключение видео и screenshare
- Tauri: захват экрана работает через WebRTC API нативно, проверить permissions
- UI: грид участников, выбор активного экрана для просмотра

**Фаза 7 — Distribution и polish**
- GitHub Actions workflow: на каждый тег `v*.*.*` собирает `.msi` под Windows, генерирует `latest.json`, публикует в GitHub Release
- **Auto-updater настройка:**
  - Сгенерировать пару ключей `tauri signer generate`, публичный в `tauri.conf.json`, приватный в GitHub Secrets
  - Прописать в `tauri.conf.json` → `plugins.updater.endpoints` ссылку на манифест в GitHub Releases
  - Реализовать в клиенте: проверка при старте + раз в час, toast-уведомление с кнопками, прогресс-бар скачивания, страница «О программе» с changelog и ручной проверкой
  - Тестовый прогон: собрать `0.0.1` → установить → собрать `0.0.2` → опубликовать → убедиться что апдейт прилетел
- Настройки пользователя (аватар, никнейм, статус)
- Production `docker-compose.prod.yml`, инструкция по деплою на Hetzner
- Бэкапы БД (pg_dump в cron)
- Базовый мониторинг (uptime kuma или просто healthcheck)
- Гайд: «как раздать установщик друзьям и как они подключатся»
- **Гайд по релизам для меня:** как делать новый релиз (`git tag v1.0.1 && git push --tags`), что проверить перед публикацией, как откатить если что-то сломалось
