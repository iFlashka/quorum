# ADR-0002: Архитектура auto-update — GitHub Releases + Tauri signer

- **Дата:** 2026-05-06
- **Статус:** Accepted (реализуется в фазе 7, заложен в фазу 0)

## Контекст

Приложение распространяется как `.msi` инсталлятор для Windows, друзья ставят его и должны автоматически получать обновления. Варианты:

1. **Свой update-сервер** — пишем endpoint, отдаёт `latest.json`. Контроль над поведением, плюс лишний сервис.
2. **GitHub Releases как хранилище + манифест в Release Assets** — `tauri-plugin-updater` поддерживает это из коробки, бесплатно.
3. **TauriBundler S3 + CloudFront** — оверкил для пет-проекта.
4. **Не делать auto-update, рассылать `.msi` руками** — отвергаем, в PROJECT.md явно указано как требование.

## Решение

**GitHub Releases** как источник обновлений + **Tauri signer** для криптографической подписи манифеста.

### Релизный флоу (фаза 7)

1. `git tag v0.X.Y && git push --tags` — триггер GitHub Actions workflow.
2. Workflow собирает `.msi` (и опционально `.exe`) под Windows.
3. Подписывает артефакты приватным ключом из `secrets.TAURI_SIGNING_PRIVATE_KEY`.
4. Создаёт GitHub Release с тегом `v0.X.Y`, прикладывает `.msi` и `latest.json` (генерится утилитой `tauri-action`).
5. CI отдельно проверяет, что версия в `apps/desktop/package.json` совпадает с версией в `apps/desktop/src-tauri/tauri.conf.json` — иначе fail.

### Клиент (фаза 7)

- `tauri.conf.json` → `plugins.updater.endpoints` указывает на `https://github.com/OWNER/REPO/releases/latest/download/latest.json`.
- При запуске + раз в час `@tauri-apps/plugin-updater` дёргает endpoint, сравнивает версии.
- Toast `«Доступно X.Y.Z»` через sonner с кнопками `Сейчас` / `Позже`.
- Скачивание в фоне с прогресс-баром.
- Применение — после клика «Сейчас» либо при следующем запуске если выбрано «Позже».
- Страница «О программе» с changelog последних N версий и кнопкой ручной проверки.

## Последствия

**+ Плюсы:**
- Бесплатно, без своих серверов.
- Tauri-нативная цепочка доверия: клиент проверяет подпись `latest.json` своим встроенным public-ключом → прогрузка `.msi` поверх HTTPS → проверка подписи `.msi` → применение.
- Откат: достаточно опубликовать новый Release с meньшей semver — клиент его не подхватит, но я могу выпустить `0.X.Y+1` с реверт-фиксом и он прилетит как обычный апдейт.

**− Минусы:**
- Привязка к GitHub. Если GitHub Releases станет недоступен — апдейтов не будет. Приемлемо для пет-проекта.
- **Потеря приватного ключа = катастрофа.** Все клиенты с зашитым public-ключом не примут обновления, подписанные новым ключом. Решение: бэкапить ключ в 2+ места (password manager + офлайн-копия) сразу после генерации.
- **Public-ключ зашит в сборку.** Чтобы поменять — нужно собрать новую версию со старым приватным ключом и встроенным новым public, дождаться чтобы все её установили, и только потом начать подписывать новым приватным. На пет-проекте проще не менять ключ.

## Что закладываем уже в фазе 0

- Версия `0.0.1` синхронно в `package.json` и `tauri.conf.json`.
- Заглушка секции `plugins.updater` в `tauri.conf.json` с пустым `pubkey` и комментарием как заполнить (`tauri signer generate`).
- Этот ADR (фиксируем решение пока контекст свежий).

Сам плагин `tauri-plugin-updater` НЕ ставим в фазе 0 — добавится в фазе 7.

## Не путать

- **Code signing** (Microsoft Authenticode) — чтобы Windows не показывал SmartScreen warning. **Не делаем** — платно ($200–400/год), для друзей приемлемо.
- **Подпись манифеста updater** (Tauri signer / Ed25519) — **обязательна**, без неё updater откажется применять обновление. Бесплатна, ключ генерим сами.
