# ADR-0004: Presence — Redis SET+TTL для состояния, Pub/Sub для broadcast

- **Дата:** 2026-05-06
- **Статус:** Accepted (реализован в фазе 2D)

## Контекст

В фазе 2 нужно показывать кто из друзей онлайн в member-list гилды. До этого статус `online/offline` приходил только из БД (поле `users.status`), что не отражает реальное состояние WebSocket-коннектов: юзер может закрыть клиент, но в БД останется «online», пока кто-то его явно не сбросит.

Требования:

1. Источник истины — наличие живого WS-коннекта, а не запись в БД.
2. Граф fan-out: я должен видеть presence только тех, с кем у меня есть общая гилда.
3. Несколько вкладок/устройств одного юзера — это всё ещё `online`. Сбрасывать в `offline` только когда отвалилась последняя сессия.
4. Crash-resilient: если процесс сервера упал без graceful shutdown, presence не должен залипнуть в `online` навсегда.
5. Готовность к горизонтальному масштабированию (хотя для 5–10 друзей оно не требуется): если когда-то будет больше одной ноды, дизайн должен это переживать без переписывания.

## Решение

**Состояние** — Redis SET с TTL. Под каждого юзера ключ `presence:user:{userId}`, значения — `sessionId` (uuid v4 на каждый WS-коннект). На каждый `connect` идёт `SADD + EXPIRE 60s`. На `disconnect` — `SREM`; если SET опустел, ключ удаляется явно. Юзер `online` ⇔ ключ существует и непустой.

**Heartbeat** — таймер на сервере раз в 30s обновляет TTL ключа (`EXPIRE 60`). Если процесс сервера падает, через ≤ 60s ключ испаряется сам. Здесь же видна разница с heartbeat-loss таймером WS (60s без сообщений → close): они выровнены, но это два разных механизма — Redis-TTL чистит presence при крахе ноды, WS-heartbeat закрывает зависшие соединения.

**Broadcast** — Redis Pub/Sub-канал `presence:events`. На переходе `0 → ≥1` (юзер пришёл) или `≥1 → 0` (юзер ушёл) сервер `PUBLISH`-ит `{userId, status, instanceId}`. Все ноды (включая ноду-источник) подписаны на канал и при получении события смотрят, в каких гилдах состоит этот `userId`, и через локальный `EventBus.publishToGuild` доставляют `presence.update` всем своим WS-коннектам, подписанным на эти гилды. Подписчик-юзер увидит presence только если у него есть общая гилда с этим userId — это и решает требование fan-out по социальному графу.

**Initial-snapshot** — при подключении WS отправляет `ready` со срезом `presence: {userId, status}[]` для всех members гилд этого юзера. Срез строится через `EXISTS` по списку userId (один pipeline) — для типичной гилды на 10 человек это ровно один Redis-роундтрип.

**Юзер-выставляемый статус (`idle / dnd`)** — отдельная семантика, не реализуется в фазе 2D. Это будет поле `users.status` в БД с собственным broadcast'ом, добавится в фазе 7. Фаза 2D отвечает только за connection-presence (`online | offline`).

## Альтернативы которые отклонены

- **Просто in-memory Map в EventBus.** Чистый, без дополнительной инфры. Отклонён, потому что не переживает рестарт процесса (любая нода после рестарта не знает кто реально online), и потому что Redis всё равно уже в стеке.
- **Redis-key per session с pattern-scan.** `presence:session:{sessionId} → userId`, чтобы понять кто online — `KEYS presence:session:*`. Отклонён: KEYS блокирует Redis, SCAN — медленнее SET+EXISTS, сложнее посчитать «сколько активных сессий у юзера».
- **DB-update на каждый connect/disconnect.** Отклонён: write-amplification на UPDATE users.status на каждый коннект, плюс не решает мульти-сессии.
- **Только Pub/Sub без хранения состояния.** Отклонён: при опоздании клиента (зашёл после того как юзер X стал online) — не получит initial-снимок. SET даёт snapshot-источник истины, Pub/Sub — поток обновлений; и то и другое нужно.

## Последствия

**+ Плюсы:**
- Один Redis-инстанс уже есть в стеке — никаких новых зависимостей.
- Архитектура честно работает в multi-node, хотя фактически нода одна.
- TTL даёт self-healing после крашей.
- Multi-session honored бесплатно — SET автоматически считает сколько коннектов.

**− Минусы / на что обратить внимание:**
- Каждый presence broadcast → SQL на `members` (получение guildIds юзера). Для 5–10 друзей это копейки, но если когда-то будет 1000+ юзеров — стоит закэшировать `userId → guildIds[]` (например, на 30s или через инвалидацию при join/leave). Сейчас не делаю — преждевременная оптимизация.
- TTL 60s + heartbeat 30s означает максимальный лаг прокидывания «упал» в случае «незаметного» падения = ~60s. Для friends-чата приемлемо.
- Если когда-то добавим `idle/dnd`, нужно будет различать «connection-online но idle» — придётся хранить юзер-выставленный статус отдельно (БД) и совмещать с connection-presence на чтении.
