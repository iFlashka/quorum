# ADR-0001: Монорепа на pnpm workspaces

- **Дата:** 2026-05-06
- **Статус:** Accepted

## Контекст

Quorum — это сервер (`apps/server`) и десктоп-клиент (`apps/desktop`), которые делят типы и контракты API. Варианты организации:

1. **Два независимых репозитория** + опубликованный пакет общих типов в npm.
2. **Монорепа на pnpm workspaces** — `apps/server`, `apps/desktop`, `packages/shared`.
3. **Монорепа с инструментом оркестрации** (Nx, Turborepo, Lerna).
4. **Один пакет** — фронт и бэк в одной папке, без workspaces.

## Решение

Монорепа на **pnpm workspaces** без Nx/Turborepo.

`packages/shared` экспортируется напрямую из исходников TypeScript (`"main": "./src/index.ts"`), без сборочного шага — потребители (`@quorum/server`, `@quorum/desktop`) импортируют типы и Zod-схемы напрямую. Это работает потому что обе стороны под TypeScript-aware bundler-ами (`tsx` для Node, Vite для клиента).

## Последствия

**+ Плюсы:**
- Изменения в схемах типов сразу видны в обеих частях, без публикации пакета.
- Один lockfile, одна `node_modules` структура, один прогон CI на всё.
- Tooling простой: `pnpm -r --if-present test`, `pnpm --filter @quorum/server dev`.
- pnpm экономит место и быстро ставит — критично при частом переключении веток.

**− Минусы:**
- Фронтенд и бэкенд связаны общим релизным циклом — при критическом баге сервера нельзя «откатить только сервер» простым `npm install older-version`. Решение: версионировать `apps/desktop` независимо через тег `desktop-v*.*.*`.
- Без Nx/Turborepo нет кэша сборки. Для проекта на ~3 пакета это излишне — добавим, если станет больно.

## Альтернативы и почему отвергнуты

- **Nx/Turborepo** — оверкил для пет-проекта на 3 пакета. Добавляет concept-overhead и свои конфиги.
- **Два репо + опубликованный @quorum/shared** — медленный feedback loop при смене типов, нужно публиковать пакет на каждое изменение.
- **Один пакет без workspaces** — теряется чёткое разделение dependencies (фронт случайно тянет `pino`, бэк — `react`).
